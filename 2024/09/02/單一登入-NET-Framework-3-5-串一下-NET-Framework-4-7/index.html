  1. SSO
    使用一些SSO協定，自架 IdentityServer 或 Azure Active Directory 之類的服務一起使用。
    每個網站在用戶登入後，都會向 SSO 提供者進行驗證，以便獲取 User.Identity.Name，我自己幫他亂結案為主從式。

  2. Forms Authentication
    ASP.NET 的表單身份驗證（Forms Authentication)，用相同的machineKey，遊走在各站之間溝通。

  3. 使用共享資料庫或 API
    將他們的登入資訊儲存到共享資料庫或透過 API 傳遞給其他網站。
    其他網站在接收到用戶的請求時，可以從共享資料庫或 API 中讀取登入狀態
    將用戶的 User.Identity.Name 與 session 資訊等存到資料庫後， 再用其內容來確認用戶的登入狀態。感覺這個玩法和第1條路有異曲同工的味道。

  4. 使用跨域 Cookie
    基本上Forms Authentication也算是這個的基本型，只是這裡的加密解密需要自己搞。
    當然也可以反過來說跨域 Cookie是Forms Authentication的變型!(ps：我自己亂結案的觀點)

  5. 令牌
    在查資料時 有一點點掃過JWT的關鍵字，然後在多個網站之間傳遞這個令牌來驗證用戶身份。
    這可以透過 URL、Header 或者 Local Storage 來傳遞，
    但資料相對的較少，沒有很認真往這條路往下走查。
    必竟查到這裡，內心有點還滿確定的大約就是1-3選方向選一個來做。

解決方案：
流程實際上滿簡易的

  1. 走公司本來的登入羅輯，如果過了我就後面+發卡動作(CreateTicket)
    發卡動作：拿到使用者資訊→加密→Cookies.Add(authCookie)
  2. 在web.config中+認卡動作
    認卡動作：拿到authCookie→解密→解的過就去塞context.User
    (因為我的環境中本來認定的登入是認context.User，如果你的情況是其他 如session等，反正該塞什麼就塞什麼)
  3. 登出流程，導到登出頁將Cookies設定過期
    (這一段我還在思考中，必竟原環境中的登出五花八門，怎樣才能搞到最不痛或無痛…真的是個好問題)
